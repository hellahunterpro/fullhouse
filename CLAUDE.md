## What this project is

Full House: a play-money virtual-chip casino as a Telegram Mini App. Current work is
**Stage 0**, defined task-by-task in `docs/STAGE-0-PLAN.md`.

## Hard guard (non-negotiable)

- **No real money, ever, in this build.** Chips are never purchasable (including via
  Telegram Stars or TON), never cashable, never tradeable for value. Do not add,
  scaffold, or stub any payment, cashout, crypto, or conversion path. If a task ever
  seems to require one, stop and ask.
- **Server-authoritative.** The server computes every outcome. Never trust a
  client-supplied outcome or balance.

## Autonomous build protocol

Work `docs/STAGE-0-PLAN.md` from the top. For each task:

1. Implement it to the stated **Definition of Done**.
2. Run the task's **Verification** steps. Fix failures yourself.
3. Commit (Conventional Commit, **prefixed `[skip ci]`**) and **push**.
4. Move to the next task **without asking for confirmation.**

Keep going through tasks unattended. Do **not** stop for trivial confirmations,
phrasing choices, or "should I continue" — continue.

**STOP and surface a short summary instead of proceeding** only when one of these is
true:

- You are about to **deploy** (`wrangler deploy`, `wrangler pages deploy`), create
  remote/cloud resources, change any platform setting, or run a command against a
  `--remote` D1 database. Never do these autonomously. (Pushing to GitHub is fine —
  see below.)
- You reach a **correctness-critical checkpoint** (marked `CHECKPOINT` in the plan:
  wallet atomicity, RNG fairness). Present the approach in a few lines and wait for a
  human to confirm before building on top of it.
- A command is **destructive** (mass delete, history rewrite, anything touching files
  outside the repo).
- The **same verification fails three times** and you cannot resolve it — report what
  failed and what you tried, then stop.

Everything else: proceed.

## Repository conventions

- **Language:** all repository-visible text (code, comments, commit messages,
  `ARCHITECTURE.md`, `README.md`, the build plan) is in **English**.
- **No AI fingerprint in the repo.** No "generated by", no assistant mentions, no
  co-author trailers.
- **Commits:** Conventional Commits — `feat:`, `fix:`, `refactor:`, `chore:`,
  `docs:`, `test:`. Every commit message is prefixed with `[skip ci]` by default.
  Remove the prefix only when a human explicitly instructs a deploy.
- **Comments:** only where they explain non-obvious logic.
- Commit per completed task, small and focused; keep the history clean.

## Deploy discipline

- **Never deploy.** Deployment is a human action triggered by an explicit instruction
  ("deploy" / "ship" / "проверь на телефоне"). Not via the Cloudflare dashboard, not
  via MCP.
- Pushing to GitHub after each task is expected. Every commit carries `[skip ci]`, so
  pushes never trigger a deploy.
- Do not configure cloud credentials or `.env` / `.dev.vars` with real values. Local
  development uses local D1 (`--local`).

## Local verification

- `npm install`, then `npm run dev` (web) and the Worker dev server boot cleanly.
- `npm run typecheck` and `npm run lint` pass; `npm test` passes.
- Local D1 migrations apply with `--local`.

## Definition of "Stage 0 done"

A real, playable dice game running locally end-to-end: a Telegram-authenticated user
with a starting chip balance places a dice bet; the server validates → commits/reveals
an RNG seed → resolves (pure) → moves chips atomically through the wallet → writes an
audit event → returns the result with a verifiable fairness proof. Full test suite
green, neutral English `README.md`, clean git history. No deploy.
